Network Panel - Workshop

Instructions: Open the Network tab in DevTools before clicking any button. Keep it open throughout the session.

1. Recording, Preserve Log, and Clear

The Network panel records all requests automatically when open. Preserve log keeps previous requests visible after a page reload or navigation. Clear removes all rows without stopping recording.

Click "Make a Request" and watch a row appear in the Network panel.

Try: enable Preserve log in the Network toolbar, click Reload, and confirm previous requests are still visible. Then disable it and reload again — the list clears on navigation.

2. Request Table Columns

Each row in the Network table has key columns. Click the buttons below and observe what each column shows.

Responses will appear here.

Columns to observe: Name (URL), Status (200/404), Type (fetch/xhr), Initiator (script line), Size (bytes transferred), Time (total duration).

3. Request Detail Tabs

Click a request in the Network panel to open its detail pane. Each tab below covers a different part of the request.

3.1 Headers

Click the request in Network → Headers tab to see Request Headers, Response Headers, and General info.

3.2 Payload

Click the request → Payload tab to see the JSON body sent to the server.

3.3 Preview

Click the request → Preview tab to see the formatted JSON tree.

3.4 Response

Click the request → Response tab to see the raw response body text.

3.5 Timing

Click the request → Timing tab to see Stalled, DNS, Connection, SSL, TTFB, and Content Download breakdown.

4. XHR vs Fetch

Both XHR and Fetch appear under the Fetch/XHR filter but are different APIs. Compare them side by side in the Network panel.

Both requests will appear in the Network panel under the Fetch/XHR filter.

In the Network panel: click the Fetch/XHR filter button to isolate these requests. Check the Type column — Fetch shows as fetch, XHR shows as xhr.

5. Filtering by Resource Type

The filter bar at the top of the Network panel lets you show only certain resource types. Trigger multiple types and then filter them.

Load a few resources, then use the type filter buttons in the Network panel toolbar.

Filter buttons: All · Fetch/XHR · JS · CSS · Img · Media · Font · Doc · WS · Other. Click multiple buttons to combine filters.

6. Advanced Filter Queries

Type filter expressions directly into the Network filter input to narrow requests by status, size, domain, or header.

Trigger a few requests, then try the filter queries listed below.

Try these in the Network filter input:

QueryWhat it shows
status-code:404Only 404 responses
status-code:200Only successful responses
resource-type:fetchOnly Fetch API requests
resource-type:xhrOnly XHR requests
larger-than:1kResponses larger than 1 KB
larger-than:10kResponses larger than 10 KB
domain:jsonplaceholder.typicode.comRequests to this domain only
-domain:jsonplaceholder.typicode.comExclude that domain (negative filter)

7. Network Throttling and Offline Mode

Use the throttle dropdown in the Network toolbar to simulate slow connections and test how the page behaves under degraded conditions.

Set throttle before clicking, then compare request timings.

How to throttle:

Network toolbar → throttle dropdown (default: No throttling) → choose a profile:

ProfileDownloadLatency
Slow 3G400 KB/s400 ms
Fast 3G1.6 MB/s50 ms
Offline0—

Try Offline mode to see requests fail. Remember to set it back to No throttling after.

8. Request Blocking

Block specific URLs or patterns to test how the application handles missing resources or failed API calls.

Click a button. Then follow the setup below and click again to see the request blocked.

How to block a request:

  1. Network panel → three-dot menu (⋮) → Request blocking
  2. Check Enable request blocking
  3. Click + and enter a pattern: todos/5 to block the API, or placeholder to block the image
  4. Click the button again — the request will fail with a net::ERR_BLOCKED_BY_CLIENT error
  5. The app output shows the error it received

9. Timing Analysis and TTFB

The Timing tab breaks down every phase of a request. TTFB (Time to First Byte) is the time from the request being sent until the first byte of the response arrives — it reflects server processing time.

Click a request in the Network panel, then open the Timing tab.

Timing breakdown:

PhaseWhat it means
Queued / StalledBrowser waiting before it could send the request
DNS LookupResolving the domain name to an IP address
Initial connectionTCP handshake with the server
SSLTLS/HTTPS negotiation
Request sentTime to upload the request headers and body
Waiting (TTFB)Server processing time — the key metric for slow APIs
Content downloadTime to receive the full response body

10. Copy as cURL

Any request in the Network panel can be exported as a curl command to replay from a terminal, share as a reproduction, or test with modified headers.

After the request appears in Network, right-click it → Copy → Copy as cURL (bash).

The copied command includes the full URL, method, headers, and body. Paste it into a Linux/macOS terminal to replay the exact same request outside the browser.

11. Disable Cache

When DevTools is open, you can disable the browser cache so every reload fetches fresh resources from the server. This is essential when testing changes that might be masked by a cached response.

Click twice. The second response may show "from disk cache" in the Size column.

Enable: Network toolbar → Settings gear (⚙) → Disable cache (while DevTools is open). Then reload — every request hits the network, and the Size column shows actual transferred bytes.

12. Waterfall — Parallel vs Sequential Loading

The waterfall column visualises when each request started and how long it took. Parallel requests run simultaneously and finish faster than sequential ones.

Compare the total time between parallel and sequential loading in the waterfall.

After running both, look at the waterfall column in the Network panel. Parallel requests overlap; sequential ones form a chain where each starts after the previous finishes.
