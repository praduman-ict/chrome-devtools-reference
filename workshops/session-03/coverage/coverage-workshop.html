1. Opening Coverage and Recording Modes

Coverage has two recording modes. Understanding which to use changes what you see.

Mode 1 — Reload and record (📄 icon):

Reloads the page and records from the very first line that executes. Best for seeing what code runs at page load before any user interaction.

Mode 2 — Start recording without reloading (⏺ icon):

Records while you interact. Code that only runs on button clicks or events appears here. Use this after page load to see how much additional code your interactions cover.

3. Conditional Code Paths

If you only trigger the if branch, the else branch lines stay red. Only by clicking both buttons do you cover 100% of the conditional.

Click one button, check Coverage. Then click the other and re-check.

Real-world implication: high coverage % does not mean all logic is tested — only that the lines were executed. A test suite that only exercises the happy path will miss the else branches.

4. Feature Flag Code

All three feature flags below are set to false. The functions behind them (enableDarkMode, enableBetaSearch, enableAdminPanel) are defined in the file but can never be reached — they will always be red in Coverage.

The output will always be "all features disabled" — the flag code is unreachable.

This is one of the most common sources of unused JS in production apps: shipped feature-flag code that is never enabled. Coverage surfaces it immediately.

5. CSS Coverage

Coverage tracks CSS rules too. Rules applied to elements on the page turn green. Rules that exist in the stylesheet but match no element stay red.
